## P0: Remove stale concepts and terminology


- [x] Remove or rename `src/runtime/session_registry.zig`.
  It is not a registry anymore. It validates/generates GUIDs and holds
  `default_ssh_port`. Split this into something like `core/guid.zig` plus a
  transport/config home for the default SSH port.

- [x] Audit all GUID surfaces after sesshmux removal.
  Current code still has session and proxy GUID helpers, which may be fine, but
  the old "registry", short-guid, alias, route, and reattach mental model should
  not remain in names or comments.

- [x] Remove stale "attachment" terminology.
  `attached_client` is acceptable if it means the visible client currently
  connected to a terminal session. Other uses of "attachment" should become
  "attached client", "client connection", or a more specific term.

- [x] Rename stale `banner` wording in tests/docs to `overlay`.
  Examples remain in `tests/ssh_reconnect_tmux_harness.py` and some harness test
  names/assertion messages. The product term is overlay.

- [x] Audit `raw` terminology.
  Keep it where it means terminal raw mode or uninterpreted byte stream. Remove
  it where it is an old filter-level name or where `unhygienic` is the actual
  user-facing concept.

- [x] Remove or quarantine remaining `sesshmux` references if any appear after a
  full grep. There should be no sesshmux CLI, state, tombstone, route, list, or
  reattach vocabulary left.

- [x] Remove `tombstone` and old pending-kill vocabulary if any remains.
  The current cleanup model is remote-process records plus daemon cleanup; old
  tombstones are from the sesshmux design.

- [x] Revisit `compat` naming.
  `SESSH_COMPAT`, `compatModeFromEnv`, and "falling back to plain ssh without
  persistence" predate the current daemon architecture. Keep only if the mode is
  intentional and documented; otherwise delete or rename around the actual
  behavior.

- [x] Clean up `:broker:` and `:proxy:` legacy modality naming.
  `:proxy:` is gone. Bootstrapped broker startup uses the `sessh-broker` role
  name; `sessh :broker:` remains only as the documented no-bootstrap fallback
  for remotes whose PATH may expose `sessh` but not the role symlink.

- [x] Remove `:internal*:` support completely if any remains.
  Top-level arg1 should be either a public option/subcommand shape or a current
  process role fallback. No hidden legacy modality should survive.

- [x] Fix misleading proto comments for top-level payloads.

- [x] Decide whether `ClientDaemonItem.ProxyDiagnosticsOpen` still belongs in the

  It remains for process-isolated proxy mode, where a visible client diagnostics
  connection and a long-lived proxy process must rendezvous by p-guid.

- [x] If `ProxyDiagnosticsOpen` remains, rename it around what it actually does.
  It is now named around proxy diagnostics/retry routing instead of the old
  generic proxy-control side channel.

- [x] Re-evaluate `--diagnostics-guid`.
  The old `--control-guid` name is gone. Process-isolated proxy mode still needs
  an internal p-guid rendezvous between the visible diagnostics connection and
  the long-lived proxy process.

- [x] Clarify `SshTransportAcquire`.
  It currently includes transport identity, visible-client environment,
  cleanup owner pid/start time, and isolation mode. Some of those are not
  actually part of acquiring/reusing the SSH transport. Either rename the
  message to indicate it is an acquisition plus first-use context, or split
  transport-key fields from per-open fields.

- [x] Add tests that prove the pooled transport reuse key is explicit and minimal.
  The request includes host, ssh options, bootstrap, ssh_auth_sock, and IPQoS.
  The code should make clear which fields affect reuse and which only affect
  child ssh setup. Test both reuse and non-reuse cases so hidden reuse-key logic
  cannot creep back into `ssh.zig`.

- [x] Audit backpressure fields in `MuxStreamFrame`.
  `receive_window_bytes` and `Ack` document per-stream credit. Add tests that
  prove the implementation enforces fixed-size per-stream buffers and fairness,
  or mark/trim the fields until enforcement exists. A protocol field that looks
  enforced but is not enforced is dangerous.

- [x] Make mux stream fairness explicit.
  When several streams are writable, a single busy proxy stream should not
  starve terminal data or another proxy stream. The daemon should have a simple
  scheduling policy, and tests should cover it once backpressure is enforced.

- [x] Add tests that prove `MuxStreamFrame.Reset` is used only for abrupt
  termination.
  It is documented as TCP RST / HTTP/2 RST_STREAM. Keep it only for abrupt
  termination. Graceful terminal session end should be `SessionEnded` followed
  by mux EOF.

- [x] Clarify `RemoteProcessIdentity.guid`.
  The cleanup JSON filename should not duplicate the GUID. The protocol may
  still need a GUID to identify remote resources over the wire. Keep the
  boundary clear so the on-disk format stays simple.

- [x] Make cleanup protocol names match cleanup docs.
  `RemoteProcessStarted`, `RemoteProcessRecorded`, and cleanup request/response

  "stream type" terminology where terminal-emulator work is not just a stream.

- [x] Consider grouping protocol helper names by top-level frame group.
  Terminal-emulator helpers now avoid old "TeStream" wording.
  Prefer names matching current proto groups, e.g. terminal-emulator item,

- [x] Stop calling generated proto internals from too many places.
  Centralize common encode/decode/send paths in protocol helpers so business
  logic does not need to know the exact nested generated type names everywhere.
  Generated oneof type names are now confined to `protocol/typed_send.zig`,
  which exports protocol-level aliases for production modules.

## P0: Nonblocking IO and dispatcher cleanup


- [x] Make "one dispatcher per process" true in production code.
  Direct `poll(2)` loops remain in `session/client_ui.zig`,
  `session/attached_client.zig`, `session/runtime.zig`, `stream/runtime.zig`,
  `daemon/mod.zig`, `transport/bootstrap.zig`, `transport/plain_ssh.zig`, and
  `tty/terminal.zig`. Some are legitimate process event loops, but the long-term
  shape should be explicit: a process either owns one dispatcher or is a small
  foreground compatibility path with a documented reason.
  Sesshd owns one `Dispatcher`. Remaining direct poll loops are either
  `PROCESS_EVENT_LOOP` loops for a whole foreground/helper process or
  `BLOCKING_POLL` foreground setup/probe compatibility paths.

- [x] Remove production `FrameReader.readBlocking` call sites where possible.
  Remaining uses are intentionally foreground-only: daemon handshake,
  `sessh --daemon-log`, visible-client startup/reconnect waits, and
  `sessh-proxy` fd-pass acceptance. Each site is labeled with
  `BLOCKING_FRAME_READ` and explains why dispatcher ownership is not useful
  there.

- [x] Keep blocking frame reads in tests/test helpers only where practical.
  The blocking test frame reader now lives in `protocol/test_helpers.zig`.
  Production modules' tests use that shared helper or the nonblocking
  `FrameReader`/proxy-diagnostics `Reader` directly.

- [x] Audit `core/io.zig` blocking `readExact`/`writeAll`.
  They are convenient, but on a nonblocking architecture every production use
  should be intentionally bounded, foreground-only, or replaced with write
  state. This is especially important for daemon paths and pooled transports.
  The helpers now document their blocking contract. Shared daemon paths use
  dispatcher-owned readers/writers; remaining direct uses are foreground UI,
  tests, filesystem writes, process-local bootstrap/plain-ssh paths, or
  close-on-error protocol responses.

- [x] Replace ad hoc direct writes with write state where backpressure matters.
  `protocol.sendFrame*` currently writes immediately via `io.writeAll`. That is
  acceptable for small foreground setup paths, but daemon/client/stream state
  machines should not assume a whole frame can be written without blocking.
  The process-isolated proxy diagnostics router no longer writes directly to
  pooled client fds; it queues visible-client output and routes retry frames
  through the pooled client's write state.

- [x] Continue migrating frame forwarding to explicit state machines.
  `transport/frame_forwarder.zig` has `ReadBuffer`, `WriteBuffer`, and
  `FrameForwarder`; add relay-path tests that prove daemon relays use those or
  equivalent dispatcher-owned state, not blocking decode/write loops.

- [x] Audit direct PTY writes.
  The remote terminal runtime writes PTY input with `writeSomeNonBlocking` and
  a bounded pending-input queue; tests cover partial/nonblocking flush behavior.
  Remaining direct PTY writes are in foreground/plain-ssh compatibility paths,
  where no daemon dispatcher work is being starved.

- [x] Replace `waitpid` blocking where it can block process progress.
  WNOHANG checks are fine. `tty/pty_process.zig` has a blocking cleanup wait
  after PTY close; either convert it to nonblocking process cleanup or document
  why no other work can exist at that point.

- [x] Remove production sleeps and fake timers.
  Waiting should be dispatcher timers. If an intentional foreground startup wait
  remains, keep it short, bounded, and documented.

- [x] Make `daemon/startup.zig` lock/socket contention simpler.
  Blocking on the daemon startup lock is acceptable for the visible client only
  because there is no useful other work before a daemon connection exists. The
  code should make this exception obvious and bounded.

## P0: Daemon/process lifecycle cleanup


- [x] Add tests that prove daemon death messages distinguish local vs remote.
  The visible client should report local daemon connection loss differently from
  remote daemon death/transport loss.
  For `isolation-mode=process`, also test that an active connection recovers
  after local daemon death and after remote daemon death. That recovery property
  is the point of keeping the proxy stream outside the daemon process.

- [x] Add tests that prove terminal restoration happens before error output.
  If the local daemon dies while the visible client owns a TTY, restore local
  terminal state before printing `sessh: local daemon connection lost` or any
  similar message.

- [x] Add tests that prove client death cleans up terminal and proxy resources.
  The local daemon should notice visible-client socket close and ask the remote
  daemon to hang up the terminal process or close the proxy resource. The same
  rule should apply whether the visible client exits normally, is killed, or the
  local machine later reboots and cleanup falls back to persisted records.

- [x] Add tests that prove session death exits the client ssh-shaped.
  When the remote command exits, the visible client should restore local state
  and return the same exit behavior users expect from ssh.

- [x] Keep the daemon alive only for useful work.
  A daemon should shut down when it has no clients, no live pooled transport
  worth preserving, and no cleanup obligation. If cleanup state exists, the
  shutdown rule should match `docs/CLEANUP.md`. Add daemon-idle tests for
  no-client shutdown, live pooled transport retention, and pending-cleanup
  retention.

- [x] Add race tests for daemon socket/flock ownership.
  The running daemon should hold the lock that proves the namespace is live.
  Symlink updates should happen only while holding the same lock and only when
  starting/restarting the daemon.

- [x] Remove runtime `meta.json` or any persistent daemon metadata that no longer
  has a purpose.
  Sessions become unusable when the daemon dies; persistent state should be
  limited to cleanup records that survive reboot.

- [x] Keep role symlinks source-of-truth clear.
  `sesshd`, `sessh-broker`, `sessh-proxy`, `sessh-terminal-remote`, and
  `sessh-proxy-remote` symlinks should all point at the same active binary in
  the daemon namespace. Packaging should not require hardlinks. Add startup and
  install-layout tests that prove the symlinks are rewritten only under the
  daemon lock and point at the daemon's executable.

- [x] Revisit whether `sessh-broker` is still the right name.
  It is currently a foreground bridge that connects local/remote daemon
  protocol over ssh. If its behavior is exactly "remote daemon tunnel bridge",
  consider whether "broker" is still clear enough.

## P0: Cleanup-record implementation


- [x] Align cleanup config names across code and docs.
  Current names include `cleanup-wakeup-interval-hours`,
  `cleanup-retry-limit-hours`, and `disconnected-reap-hours`. Earlier design
  discussions used shorter names. Pick one public naming scheme and update
  config parser, docs, tests, errors, and user manual together.

- [x] Add cleanup-record tests that prove the implementation matches
  `docs/CLEANUP.md`.
  It says records are one JSON file per GUID under `state/procs/`, mtime is the
  recorded-at timestamp, files are atomically created and only deleted, and old
  records are abandoned after the retry limit. Code should match exactly.

- [x] Add tests that prove cleanup records are equally valid for terminal and proxy
  resources.
  A proxy resource may be "close the remote sshd localhost connection" rather
  than "close a PTY", but the durable record and cleanup request should cover
  both without stream-specific special cases in the on-disk format.

- [x] Avoid duplicate cleanup work.
  The global cleanup flock should prevent multiple daemons from scanning and
  connecting to the same stale records. Daemons with live clients should be
  preferred owners where possible, as described in the doc. Add contention tests
  that start multiple daemons and prove only one cleanup sweep acts on a record.

- [x] Make stale-record deletion obvious.
  A record older than the cleanup retry limit should be deleted/abandoned with
  a daemon log entry, not silently ignored forever. Add tests for expired-record
  deletion and the corresponding daemon-log entry.

- [x] Test reboot-style cleanup.
  A record for a dead local pid/start-time should lead to a fresh remote
  connection and cleanup request. A live local pid/start-time should be skipped.
  A mismatched remote pid/start-time should be treated as missing and delete the
  local record.

- [x] Keep `RemoteProcessRecorded` ordering strict.
  If the remote tells us its pid/start-time/socket-path and the tunnel drops
  before local durable record ack, the remote side must clean up aggressively to
  avoid orphaning.

## P1: Proxy cleanup


- [x] Make proxy fd-pass the preferred `isolation-mode=none` path.
  The short-lived `sessh-proxy` should connect to local `sesshd`, pass one end
  of a socketpair with SCM_RIGHTS, wait for explicit acceptance, pass the other
  end to OpenSSH, and exit. Add tests that prove the helper exits only after
  acceptance and that OpenSSH receives the raw fd, not framed daemon IPC.

- [x] Document and test non-fd-pass proxy mode as the `isolation-mode=process`
  path.
  In this mode the proxy functionality lives in a separate `sessh-proxy`
  process, so OpenSSH should keep talking to that process instead of receiving a
  daemon-owned fd via ProxyUseFdPass. Add tests that prove
  `isolation-mode=process` does not use fd-pass and still cleans up the proxy
  process/resource correctly. Also add recovery tests that kill the local daemon
  and the remote daemon independently while a proxy connection is active; the
  `sessh-proxy` process should reconnect through the restarted daemon path
  rather than losing the OpenSSH byte stream.

- [x] Remove long-lived proxy diagnostics side channels if fd-pass can replace them.
  `proxy_diagnostics_router` is a global router keyed by p-guid. This is
  conceptually close to the older side-channel design. Keep it only if process
  isolation still genuinely needs a separate visible-client control channel.

- [x] Rename `proxy_control` if it remains.
  The shared channel is now `proxy_diagnostics_channel` and the daemon rendezvous
  is `proxy_diagnostics_router`, matching the `ConnectionEvent`/`RetryNow`
  payloads it carries.

- [x] Revisit `sessh-proxy-remote`.
  Its process role should be clear: remote endpoint for proxy byte streams,
  not a generic proxy client. Code names should match that role.

- [x] Keep proxy bytes raw only on the raw fd.
  Framed daemon IPC should never be passed directly to OpenSSH. Tests should
  keep proving that fd-pass raw bytes echo through the passed fd and the
  `sessh-proxy` process exits.

- [x] Make proxy process diagnostics follow `docs/DIAGNOSTICS.md`.
  It should forward explicit `--diagnostics-file`, auto-use the same TTY only
  when stdin/stderr are the same TTY, and otherwise write diagnostics to stderr
  or sparse line/jsonl output as selected. Add proxy-mode diagnostics tests for
  each diagnostics file/input combination.

## P1: SSH transport and startup performance


- [x] Keep startup trace machinery gated and temporary.
  Trace logs are useful for investigation, but code should not accumulate
  permanent one-off timing logs.

- [x] Measure and minimize fast-path round trips.
  The desired pooled fast path should pipeline acquire/open/session setup as
  much as protocol correctness permits. Avoid request/ready waits that only
  exist because local code is staged awkwardly.
  Pooled startup logs report request-to-open, open-to-open-ok,
  open-ok-to-first-payload, and request-to-first-payload timing, and the SSH
  harness asserts those fields for terminal and proxy pooled starts.

- [x] Move `ssh -G` responsibility to the daemon if that simplifies the visible
  client.
  The visible client now only adapts ssh-shaped CLI into `SshTransportAcquire`.
  `transport/pooled_ssh.zig` owns `ssh -G` resolution for connection pooling
  and resolved transport identity.

- [x] Keep terminal probing local but cheap.
  Probing colors/kitty/cursor locally is fine if it avoids remote round trips.
  It should not force extra serialized startup waits.

- [x] Make first-draw alignment explicit.
  If the client restores cursor state and clears below the cursor before the
  first remote draw, that should be a small named step with tests around
  non-tty fallback and resize/repaint behavior.

- [x] Revisit pooled transport logging.
  Logs should show acquire, reuse/new connection, remote hello, stream open, and
  first payload timings at important granularity without becoming verbose spam.

- [x] Add tests for IPQoS reuse-key behavior.
  Since sessh cannot retag an existing child ssh TCP socket, IPQoS belongs in
  the reuse key. That should be visible in one small function and tests should
  prove different IPQoS values do not share a pooled transport.

- [x] Add tests for SSH_AUTH_SOCK pooling behavior.
  The current design intentionally does not include it in the reuse key. Make
  sure comments and tests cover what happens when a reused transport was created
  with a now-stale agent socket.
